세부 펀치리스트

UI1. 속도 파라미터 설정

요청: SHI 담당: 박주한

AS-IS

  • 실속도 모니터링 기능 부재.
  • 로봇 및 갠트리 속도를 UI에서 제어하기 어려움.

TO-BE

  • 실속도 모니터링 도입.
  • Speeddata 4개 필드를 모두 열람·셋업 가능.

UI2. 엑셀 데이터 제어

요청: SHI 담당: 박주한

AS-IS

  • 엑셀 불러오기 시 삭제 여부만 묻고 실제 업로드 여부와 상관없이 토치/심데이터 초기화.
  • 특정 경로 파일을 수동으로 찾아야 함.

TO-BE

  • 파일 업로드 여부에 따라 데이터 초기화.
  • 기준값 호출 버튼을 통해 지정 경로 엑셀 즉시 접근.

UI3. 심데이터 추가 칼럼 구성

요청: SHI 담당: 박주한

AS-IS

  • Ign_Move_Delay, ScrapeStart 값을 관리할 칼럼이 없음.
  • 용접 전/후 처리 시간을 세부 조정 불가.

TO-BE

  • 해당 컬럼을 심데이터 테이블에 추가.
  • 내부 DB 및 RAPID에 전달하여 로봇이 시간제어 가능.

UI4. 편의성 및 가시성 개선

요청: SHI 담당: 박주한

AS-IS

  • 체크박스가 작고 클릭 정밀도가 낮음.
  • 컬럼명이 셀보다 좁아 텍스트가 잘림.

TO-BE

  • 해당 셀을 클릭하면 체크박스 선택/해제.
  • 칼럼 폭 조정 및 텍스트 래핑으로 가시성 확보.

로직5. 아크 에러 시 RAPID 제어

요청: SHI 담당: 박주한

AS-IS

  • 아크 에러 발생 시 로봇 자체 StopProcess, ExitCycle 로직에 의존.
  • 운영SW 관점에서 일관된 복구 시나리오가 없음.

TO-BE

  • 운영SW가 RAPID를 직접 제어하여 Stop → PP to Main → Run 순으로 복구.
  • 아크 에러 이후 동일 시나리오로 재시작 가능.

로직6. 계측 홈 / 기본 Z홈 구분

요청: SHI 담당: 박주한

AS-IS

  • 기본 Z홈이 사이클 타임 개선 요구로 수정되어 계측 카메라를 가리는 자세.
  • 계측 홈과 기본 Z홈을 구분할 수 없어 추가 동작 구성 불가.

TO-BE

  • 기본 Z홈과 계측 Z홈을 구분 (예: 커맨드 102 / 110).
  • 계측 Z홈은 관제SW에서 RequestReleaseRobotArm RPC로 명시적으로 호출.
